\subsubsection{\texorpdfstring{S\+D\+L\+\_\+\+Clear\+Queued\+Audio()}{SDL\_ClearQueuedAudio()}}
{\footnotesize\ttfamily \mbox{\hyperlink{begin__code_8h_aa4c7a931f4a968f818b2a1b10a432185}{D\+E\+C\+L\+S\+P\+EC}} \mbox{\hyperlink{_s_d_l__opengles2__gl2ext_8h_ae5d8fa23ad07c48bb609509eae494c95}{void}} \mbox{\hyperlink{begin__code_8h_a81faf4ba0455dc75f2e0507eddb79401}{S\+D\+L\+C\+A\+LL}} S\+D\+L\+\_\+\+Clear\+Queued\+Audio (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{_s_d_l__audio_8h_a677d9d2a6f77a7502a0dc8b45ad211bb}{S\+D\+L\+\_\+\+Audio\+Device\+ID}}}]{dev }\end{DoxyParamCaption})}

Drop any queued audio data. For playback devices, this is any queued data still waiting to be submitted to the hardware. For capture devices, this is any data that was queued by the device that hasn\textquotesingle{}t yet been dequeued by the application.

Immediately after this call, \mbox{\hyperlink{_s_d_l__audio_8h_a11a1d59fd68f8a416fa72bf14f5c9ec3}{S\+D\+L\+\_\+\+Get\+Queued\+Audio\+Size()}} will return 0. For playback devices, the hardware will start playing silence if more audio isn\textquotesingle{}t queued. Unpaused capture devices will start filling the queue again as soon as they have more data available (which, depending on the state of the hardware and the thread, could be before this function call returns!).

This will not prevent playback of queued audio that\textquotesingle{}s already been sent to the hardware, as we can not undo that, so expect there to be some fraction of a second of audio that might still be heard. This can be useful if you want to, say, drop any pending music during a level change in your game.

You may not queue audio on a device that is using an application-\/supplied callback; calling this function on such a device is always a no-\/op. You have to queue audio with \mbox{\hyperlink{_s_d_l__audio_8h_a19ef6fa2396f581aaf4d9a9512972268}{S\+D\+L\+\_\+\+Queue\+Audio()}}/\+S\+D\+L\+\_\+\+Dequeue\+Audio(), or use the audio callback, but not both.

You should not call \mbox{\hyperlink{_s_d_l__audio_8h_a7ffa484f230f6085e271a4f43e678f06}{S\+D\+L\+\_\+\+Lock\+Audio()}} on the device before clearing the queue; S\+DL handles locking internally for this function.

This function always succeeds and thus returns void.

{\footnotesize\ttfamily \mbox{\hyperlink{begin__code_8h_aa4c7a931f4a968f818b2a1b10a432185}{D\+E\+C\+L\+S\+P\+EC}} \mbox{\hyperlink{_s_d_l__stdinc_8h_add440eff171ea5f55cb00c4a9ab8672d}{Uint32}} \mbox{\hyperlink{begin__code_8h_a81faf4ba0455dc75f2e0507eddb79401}{S\+D\+L\+C\+A\+LL}} S\+D\+L\+\_\+\+Dequeue\+Audio (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{_s_d_l__audio_8h_a677d9d2a6f77a7502a0dc8b45ad211bb}{S\+D\+L\+\_\+\+Audio\+Device\+ID}}}]{dev,  }\item[{\mbox{\hyperlink{_s_d_l__opengles2__gl2ext_8h_ae5d8fa23ad07c48bb609509eae494c95}{void}} $\ast$}]{data,  }\item[{\mbox{\hyperlink{_s_d_l__stdinc_8h_add440eff171ea5f55cb00c4a9ab8672d}{Uint32}}}]{len }\end{DoxyParamCaption})}

Dequeue more audio on non-\/callback devices.

(If you are looking to queue audio for output on a non-\/callback playback device, you want \mbox{\hyperlink{_s_d_l__audio_8h_a19ef6fa2396f581aaf4d9a9512972268}{S\+D\+L\+\_\+\+Queue\+Audio()}} instead. This will always return 0 if you use it with playback devices.)

S\+DL offers two ways to retrieve audio from a capture device\+: you can either supply a callback that S\+DL triggers with some frequency as the device records more audio data, (push method), or you can supply no callback, and then S\+DL will expect you to retrieve data at regular intervals (pull method) with this function.

There are no limits on the amount of data you can queue, short of exhaustion of address space. Data from the device will keep queuing as necessary without further intervention from you. This means you will eventually run out of memory if you aren\textquotesingle{}t routinely dequeueing data.

Capture devices will not queue data when paused; if you are expecting to not need captured audio for some length of time, use \mbox{\hyperlink{_s_d_l__audio_8h_ab4c8e2a896ccf86599435af81fca928c}{S\+D\+L\+\_\+\+Pause\+Audio\+Device()}} to stop the capture device from queueing more data. This can be useful during, say, level loading times. When unpaused, capture devices will start queueing data from that point, having flushed any capturable data available while paused.

This function is thread-\/safe, but dequeueing from the same device from two threads at once does not promise which thread will dequeued data first.

You may not dequeue audio from a device that is using an application-\/supplied callback; doing so returns an error. You have to use the audio callback, or dequeue audio with this function, but not both.

You should not call \mbox{\hyperlink{_s_d_l__audio_8h_a7ffa484f230f6085e271a4f43e678f06}{S\+D\+L\+\_\+\+Lock\+Audio()}} on the device before queueing; S\+DL handles locking internally for this function.

Get the number of bytes of still-\/queued audio.

For playback device\+:

This is the number of bytes that have been queued for playback with \mbox{\hyperlink{_s_d_l__audio_8h_a19ef6fa2396f581aaf4d9a9512972268}{S\+D\+L\+\_\+\+Queue\+Audio()}}, but have not yet been sent to the hardware. This number may shrink at any time, so this only informs of pending data.

Once we\textquotesingle{}ve sent it to the hardware, this function can not decide the exact byte boundary of what has been played. It\textquotesingle{}s possible that we just gave the hardware several kilobytes right before you called this function, but it hasn\textquotesingle{}t played any of it yet, or maybe half of it, etc.

For capture devices\+:

This is the number of bytes that have been captured by the device and are waiting for you to dequeue. This number may grow at any time, so this only informs of the lower-\/bound of available data.

You may not queue audio on a device that is using an application-\/supplied callback; calling this function on such a device always returns 0. You have to queue audio with \mbox{\hyperlink{_s_d_l__audio_8h_a19ef6fa2396f581aaf4d9a9512972268}{S\+D\+L\+\_\+\+Queue\+Audio()}}/\+S\+D\+L\+\_\+\+Dequeue\+Audio(), or use the audio callback, but not both.

You should not call \mbox{\hyperlink{_s_d_l__audio_8h_a7ffa484f230f6085e271a4f43e678f06}{S\+D\+L\+\_\+\+Lock\+Audio()}} on the device before querying; S\+DL handles locking internally for this function.

\subsubsection{\texorpdfstring{S\+D\+L\+\_\+\+Open\+Audio()}{SDL\_OpenAudio()}}
{\footnotesize\ttfamily \mbox{\hyperlink{begin__code_8h_aa4c7a931f4a968f818b2a1b10a432185}{D\+E\+C\+L\+S\+P\+EC}} int \mbox{\hyperlink{begin__code_8h_a81faf4ba0455dc75f2e0507eddb79401}{S\+D\+L\+C\+A\+LL}} S\+D\+L\+\_\+\+Open\+Audio (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{struct_s_d_l___audio_spec}{S\+D\+L\+\_\+\+Audio\+Spec}} $\ast$}]{desired,  }\item[{\mbox{\hyperlink{struct_s_d_l___audio_spec}{S\+D\+L\+\_\+\+Audio\+Spec}} $\ast$}]{obtained }\end{DoxyParamCaption})}

This function opens the audio device with the desired parameters, and returns 0 if successful, placing the actual hardware parameters in the structure pointed to by {\ttfamily obtained}. If {\ttfamily obtained} is N\+U\+LL, the audio data passed to the callback function will be guaranteed to be in the requested format, and will be automatically converted to the hardware audio format if necessary. This function returns -\/1 if it failed to open the audio device, or couldn\textquotesingle{}t set up the audio thread.

When filling in the desired audio spec structure,
\begin{DoxyItemize}
\item {\ttfamily desired-\/$>$freq} should be the desired audio frequency in samples-\/per-\/ second.
\item {\ttfamily desired-\/$>$format} should be the desired audio format.
\item {\ttfamily desired-\/$>$samples} is the desired size of the audio buffer, in samples. This number should be a power of two, and may be adjusted by the audio driver to a value more suitable for the hardware. Good values seem to range between 512 and 8096 inclusive, depending on the application and C\+PU speed. Smaller values yield faster response time, but can lead to underflow if the application is doing heavy processing and cannot fill the audio buffer in time. A stereo sample consists of both right and left channels in LR ordering. Note that the number of samples is directly related to time by the following formula\+:
\begin{DoxyCode}
ms = (\mbox{\hyperlink{_s_d_l__opengl__glext_8h_aba70cd077c2c52c15358c231c6d293aa}{samples}}*1000)/freq 
\end{DoxyCode}

\item {\ttfamily desired-\/$>$size} is the size in bytes of the audio buffer, and is calculated by \mbox{\hyperlink{_s_d_l__audio_8h_a2edf30e7747584e28041b4986f89f440}{S\+D\+L\+\_\+\+Open\+Audio()}}.
\item {\ttfamily desired-\/$>$silence} is the value used to set the buffer to silence, and is calculated by \mbox{\hyperlink{_s_d_l__audio_8h_a2edf30e7747584e28041b4986f89f440}{S\+D\+L\+\_\+\+Open\+Audio()}}.
\item {\ttfamily desired-\/$>$callback} should be set to a function that will be called when the audio device is ready for more data. It is passed a pointer to the audio buffer, and the length in bytes of the audio buffer. This function usually runs in a separate thread, and so you should protect data structures that it accesses by calling \mbox{\hyperlink{_s_d_l__audio_8h_a7ffa484f230f6085e271a4f43e678f06}{S\+D\+L\+\_\+\+Lock\+Audio()}} and \mbox{\hyperlink{_s_d_l__audio_8h_a76d8179bee14dad268e65378058d897d}{S\+D\+L\+\_\+\+Unlock\+Audio()}} in your code. Alternately, you may pass a N\+U\+LL pointer here, and call \mbox{\hyperlink{_s_d_l__audio_8h_a19ef6fa2396f581aaf4d9a9512972268}{S\+D\+L\+\_\+\+Queue\+Audio()}} with some frequency, to queue more audio samples to be played (or for capture devices, call \mbox{\hyperlink{_s_d_l__audio_8h_a6ed6333b8b294eafb6a9435fd8ff89fa}{S\+D\+L\+\_\+\+Dequeue\+Audio()}} with some frequency, to obtain audio samples).
\item {\ttfamily desired-\/$>$userdata} is passed as the first parameter to your callback function. If you passed a N\+U\+LL callback, this value is ignored.
\end{DoxyItemize}

The audio device starts out playing silence when it\textquotesingle{}s opened, and should be enabled for playing by calling {\ttfamily S\+D\+L\+\_\+\+Pause\+Audio(0)} when you are ready for your audio callback function to be called. Since the audio driver may modify the requested size of the audio buffer, you should allocate any local mixing buffers after you open the audio device. \mbox{\Hypertarget{_s_d_l__audio_8h_af54523e415de7c7fdd0d48a76ab58bb3}\label{_s_d_l__audio_8h_af54523e415de7c7fdd0d48a76ab58bb3}}

{\footnotesize\ttfamily \mbox{\hyperlink{begin__code_8h_aa4c7a931f4a968f818b2a1b10a432185}{D\+E\+C\+L\+S\+P\+EC}} int \mbox{\hyperlink{begin__code_8h_a81faf4ba0455dc75f2e0507eddb79401}{S\+D\+L\+C\+A\+LL}} S\+D\+L\+\_\+\+Queue\+Audio (\begin{DoxyParamCaption}\item[{\mbox{\hyperlink{_s_d_l__audio_8h_a677d9d2a6f77a7502a0dc8b45ad211bb}{S\+D\+L\+\_\+\+Audio\+Device\+ID}}}]{dev,  }\item[{const \mbox{\hyperlink{_s_d_l__opengles2__gl2ext_8h_ae5d8fa23ad07c48bb609509eae494c95}{void}} $\ast$}]{data,  }\item[{\mbox{\hyperlink{_s_d_l__stdinc_8h_add440eff171ea5f55cb00c4a9ab8672d}{Uint32}}}]{len }\end{DoxyParamCaption})}

Queue more audio on non-\/callback devices.

(If you are looking to retrieve queued audio from a non-\/callback capture device, you want \mbox{\hyperlink{_s_d_l__audio_8h_a6ed6333b8b294eafb6a9435fd8ff89fa}{S\+D\+L\+\_\+\+Dequeue\+Audio()}} instead. This will return -\/1 to signify an error if you use it with capture devices.)

S\+DL offers two ways to feed audio to the device\+: you can either supply a callback that S\+DL triggers with some frequency to obtain more audio (pull method), or you can supply no callback, and then S\+DL will expect you to supply data at regular intervals (push method) with this function.

There are no limits on the amount of data you can queue, short of exhaustion of address space. Queued data will drain to the device as necessary without further intervention from you. If the device needs audio but there is not enough queued, it will play silence to make up the difference. This means you will have skips in your audio playback if you aren\textquotesingle{}t routinely queueing sufficient data.

This function copies the supplied data, so you are safe to free it when the function returns. This function is thread-\/safe, but queueing to the same device from two threads at once does not promise which buffer will be queued first.

You may not queue audio on a device that is using an application-\/supplied callback; doing so returns an error. You have to use the audio callback or queue audio with this function, but not both.

You should not call \mbox{\hyperlink{_s_d_l__audio_8h_a7ffa484f230f6085e271a4f43e678f06}{S\+D\+L\+\_\+\+Lock\+Audio()}} on the device before queueing; S\+DL handles locking internally for this function.
